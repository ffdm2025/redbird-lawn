<task type="auto">
  <name>Task 1: Lighthouse audit against production build</name>
  <files></files>
  <action>
    Run a local production build and preview server, then execute Lighthouse against it.

    Step 1 — Build and preview:
    ```
    npm run build
    npm run preview
    ```
    The preview server starts at http://localhost:4321 by default.

    Step 2 — Run Lighthouse CLI (install if needed):
    ```
    npx lighthouse http://localhost:4321 --output=json --output-path=./lighthouse-report.json --chrome-flags="--headless" --preset=desktop
    npx lighthouse http://localhost:4321 --output=json --output-path=./lighthouse-mobile.json --chrome-flags="--headless"
    ```
    (Default Lighthouse preset is mobile; --preset=desktop for desktop audit)

    Step 3 — Extract scores from the JSON report:
    ```
    node -e "
    const r = JSON.parse(require('fs').readFileSync('./lighthouse-mobile.json'));
    const cats = r.categories;
    console.log('Performance:', Math.round(cats.performance.score * 100));
    console.log('SEO:', Math.round(cats.seo.score * 100));
    console.log('Accessibility:', Math.round(cats.accessibility.score * 100));
    console.log('Best Practices:', Math.round(cats['best-practices'].score * 100));
    "
    ```

    Step 4 — Check bundle sizes from dist/:
    ```
    du -sh dist/ && find dist/ -name "*.js" | xargs wc -c | sort -rn | head -10 && find dist/ -name "*.css" | xargs wc -c | sort -rn | head -5
    ```

    Target scores (mobile):
    - Performance: 95+ (if below 95, investigate top opportunities in report)
    - SEO: 100
    - Accessibility: 95+
    - Best Practices: 100

    If any score misses target, read the specific audit failures from the JSON and fix the issues (common: missing alt text, color contrast, missing meta viewport, render-blocking resources). Re-run Lighthouse after each fix.

    Common accessibility fixes if needed:
    - Missing alt on images: add descriptive alt text
    - Color contrast failures: check text on colored backgrounds in DevTools
    - Form label associations: verify `htmlFor` matches input `id` in ContactForm

    Common Best Practices failures if needed:
    - HTTPS warnings only appear in production — ignore in local audit
    - Deprecated APIs: check console for warnings
    - CSP: headers only apply in Cloudflare Pages deployment, not local preview

    Document final scores before proceeding to checkpoint.
  </action>
  <verify>
    <automated>node -e "const r = JSON.parse(require('fs').readFileSync('./lighthouse-mobile.json')); const c = r.categories; const scores = {perf: Math.round(c.performance.score*100), seo: Math.round(c.seo.score*100), a11y: Math.round(c.accessibility.score*100), bp: Math.round(c['best-practices'].score*100)}; console.log(JSON.stringify(scores)); if(scores.perf < 95 || scores.seo < 100 || scores.a11y < 95 || scores.bp < 100) { console.error('SCORES BELOW TARGET'); process.exit(1); }"</automated>

    - Accessibility: ___/100 (need 95+)
    - Best Practices: ___/100 (need 100)

    Step 2 — Deploy to Cloudflare Pages:
    a. Push current code to GitHub (main branch)
    b. Go to Cloudflare Dashboard → Workers & Pages → Create → Pages → Connect to Git
    c. Select the GitHub repo → select main branch
    d. Build settings:
       - Framework preset: Astro
       - Build command: npm run build
       - Build output directory: dist
    e. Environment variables → Add: NODE_VERSION = 22
    f. Click "Save and Deploy"
    g. Wait for build to complete — confirm green checkmark
    h. Note the *.pages.dev URL

    Step 3 — Disable Auto Minify for HTML:
    - Cloudflare Dashboard → your domain (redbirdlawnservice.com) → Speed → Optimization
    - Content Optimization → Auto Minify → UNCHECK HTML (leave JS and CSS checked if present)
    - This prevents Cloudflare from corrupting Astro's React hydration payload

    Step 4 — Add custom domain:
    - Cloudflare Pages project → Custom domains → Set up a custom domain
    - Enter: redbirdlawnservice.com
    - Cloudflare will auto-configure DNS (CNAME pointing to *.pages.dev)
    - Wait for SSL certificate provisioning (usually 1-5 minutes)

    Step 5 — Verify production site:
    - Visit https://redbirdlawnservice.com — confirm page loads with HTTPS padlock
    - Open DevTools → Network tab → reload → confirm Total transferred < 500KB
    - Click "Get a Free Quote" → confirm form scrolls into view and renders
    - Click the phone number → confirm tel: link fires
    - On mobile (or DevTools mobile emulator): confirm sticky header with phone number visible
    - Open DevTools → Console → confirm zero red errors

    Step 6 — TTFB check:
    - In DevTools Network tab, click on the document request → Timing tab
    - "Waiting (TTFB)" should be under 100ms (Cloudflare edge is fast — expect 20-50ms)
